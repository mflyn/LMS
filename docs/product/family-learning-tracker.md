# 家庭版成长跟踪产品需求与功能边界

**Document status:** APPROVED
**Baseline candidate:** FGT-MVP-1
**Last reviewed:** 2026-06-18
**Change control:** Semantic changes require a review record and traceability update.

## 1. 背景与定位

当前项目原本按照“学校级学习管理系统”规划，覆盖学生管理、作业管理、学习资源、数据分析、家校互动、Web 端和移动端等能力。现阶段项目仍处于半成品状态，功能面较广，但工程闭环、前后端联调、测试稳定性和生产化能力仍需收敛。

本阶段将产品目标调整为：**先做一款面向家庭的成长跟踪软件，帮助家长和孩子共同管理每日成长任务，覆盖课内学习、课外阅读、体育锻炼、艺术审美、劳动实践、品德习惯等内容，记录过程、复盘薄弱点并形成正向激励；后续在产品稳定后，再扩展到家教、小班课或学校场景。**

家庭版第一阶段不以“教务管理”或“学校 LMS”为目标，而以“家庭成长记录 + 成长任务跟踪 + 智育错题复盘 + 亲子反馈 + 正向激励 + 德智体美劳均衡发展”为核心。

## 2. 产品目标

### 2.1 第一阶段目标

第一阶段目标是打通一个稳定、轻量、可演示、可持续迭代的家庭成长跟踪闭环：

1. 家长创建家庭账号和孩子档案。
2. 家长为孩子设置每日或每周成长任务，覆盖德智体美劳多个维度。
3. 孩子查看任务并记录完成情况。
4. 家长确认完成情况，并补充反馈。
5. 家长或孩子记录错题、锻炼、阅读、艺术、劳动、习惯等过程证据和复盘提醒。
6. 系统沉淀每日成长记录和每周成长报告。
7. 家长和孩子根据复盘结果调整下一阶段计划。

### 2.2 一句话定位

> 一款帮助家长和孩子共同管理每日成长任务，围绕德智体美劳记录过程、复盘问题并形成正向激励的家庭成长跟踪工具。

### 2.3 非目标

第一阶段不做完整学校管理系统，不做在线教育内容平台，不做复杂题库系统，不做运动健康医疗平台，也不做高并发、多租户、强组织架构的学校级 SaaS。

## 3. 用户角色

### 3.1 第一阶段角色

| 角色 | 说明 | 核心诉求 |
| --- | --- | --- |
| 家长 | 家庭成长管理的主要使用者和数据维护者 | 了解孩子学习、运动、阅读、艺术、劳动和品德习惯情况，布置任务、记录问题、查看趋势、及时介入 |
| 孩子 | 成长任务的执行者和自我反馈者 | 知道今天要做什么、完成后获得反馈、看到自己的进步 |

### 3.2 后续可扩展角色

| 角色 | 扩展时机 | 说明 |
| --- | --- | --- |
| 家教 / 老师 | 家庭版稳定后 | 家长邀请老师查看某个孩子，老师可布置任务或给反馈 |
| 学校教师 | 学校版阶段 | 管理班级、批量布置作业、查看班级统计 |
| 学校管理员 | 学校版阶段 | 管理组织、班级、教师和学生数据 |

## 4. 核心产品闭环

家庭版的核心闭环如下：

```text
创建孩子档案
    ↓
设置成长任务 / 成长目标
    ↓
孩子或家长记录完成情况
    ↓
沉淀每日成长记录
    ↓
记录错题、锻炼、阅读、艺术、劳动和习惯表现
    ↓
生成周报、复习提醒和成长提醒
    ↓
家长与孩子共同调整计划
```

这个闭环优先解决家庭场景中的四个问题：

1. **看得见**：家长能看到孩子每天和每周在德智体美劳各维度的投入和完成情况。
2. **记得住**：系统能沉淀任务、错题、知识点、运动、阅读、艺术、劳动、习惯和状态记录。
3. **能调整**：家长和孩子能根据趋势、错题、体能、兴趣、习惯和状态调整后续计划。
4. **不偏科**：系统能提醒家庭避免只盯课内成绩，帮助孩子保持全面发展。

## 5. 第一阶段必须完善的功能

### 5.1 家庭账号与孩子档案

#### 目标

将原学校场景下的“学生 / 班级 / 教师 / 管理员”收敛为家庭场景下的“家庭 / 家长 / 孩子”。

#### 功能清单

家长侧：

- 注册、登录、退出登录。
- 创建家庭。
- 添加孩子。
- 编辑孩子基本信息：姓名或昵称、头像、年级、学校、教材版本、兴趣、薄弱科目、运动偏好、艺术兴趣、劳动习惯和品格目标。
- 支持一个家庭管理多个孩子。
- 支持家长在多个孩子之间切换视图。

孩子侧：

- 支持孩子独立入口。
- 支持简单 PIN 码或家长授权登录。
- PIN 由家长设置或重置；连续失败必须限流，重置后旧的孩子 token 失效。
- 孩子只能查看自己的任务、成长记录、错题和奖励。

#### 暂不做

- 学校组织。
- 班级管理。
- 教师认证。
- 学籍管理。
- 学生批量导入。
- 学校管理员。

### 5.2 成长维度体系

#### 目标

将系统从“课内学习任务跟踪”升级为“德智体美劳全面成长跟踪”。所有任务、记录、目标和周报统计都应归属到一个成长维度，便于家长看到孩子发展是否均衡。

#### 成长维度

| 维度 | 说明 | 示例任务 |
| --- | --- | --- |
| 德育 | 品德、习惯、责任感、情绪和社交行为 | 主动整理书包、礼貌表达、遵守约定、情绪复盘 |
| 智育 | 课内学习、课外阅读、知识点、错题和思维训练 | 数学练习、英语阅读、科学实验、错题订正 |
| 体育 | 体能、运动技能、户外活动和健康作息 | 跳绳、跑步、游泳、篮球、早睡打卡 |
| 美育 | 音乐、美术、审美、表达和创造 | 练琴、画画、手工、观展、朗诵 |
| 劳育 | 家务、劳动技能、生活自理和服务意识 | 洗碗、整理房间、做饭准备、照顾植物 |

#### MVP 范围

第一阶段每条任务和每日记录都必须选择一个成长维度。智育可以继续使用科目、知识点和错题；体育、美育、劳育、德育先使用“活动类型 + 过程记录 + 家长反馈 + 孩子自评”的通用结构，不做复杂专业测评。

### 5.3 成长任务 / 待办清单

#### 目标

用轻量“成长任务”替代学校式“作业发布、提交、批改”，同时覆盖课内学习和课外成长活动。

#### 功能清单

家长可以创建任务：

- 成长维度：德育、智育、体育、美育、劳育。
- 领域 / 科目 / 活动类型。
- 任务标题。
- 任务类型：作业、预习、复习、阅读、背诵、练习、错题订正、体育锻炼、艺术练习、劳动实践、习惯养成、品德行为、兴趣探索。
- 截止日期。
- 预计用时。
- 目标数量或强度，例如跳绳 500 个、阅读 20 页、练琴 30 分钟、整理房间 1 次。
- 优先级。
- 是否重复：每天、每周、自定义（第二阶段能力）。
- 任务说明。
- 附件或图片，作为可选能力。

孩子可以处理任务：

- 查看今日任务。
- 标记完成。
- 填写实际用时。
- 上传完成照片。
- 记录难度感受。
- 标记是否需要家长帮助。

家长可以跟进任务：

- 确认完成。
- 补充评价。
- 调整任务。
- 查看完成率。

#### MVP 范围

第一版只需要支持：

- 今日任务。
- 本周任务。
- 成长维度。
- 领域 / 科目 / 活动类型。
- 完成 / 未完成。
- 实际用时。
- 实际数量，可选。
- 简单备注。
- 家长确认。

第一版每条任务只表示一次发生，不实现重复任务模板。每天执行的习惯由每天一条任务表示，从而保证每日完成状态和周报的 `4/7` 等统计没有歧义。重复任务模板和 occurrence 生成进入第二阶段。

### 5.4 每日成长记录

#### 目标

记录每天的学习、运动、阅读、艺术、劳动和习惯内容，以及用时、完成情况、专注状态、身体状态和情绪状态，让家庭成长过程可追踪、可复盘。

#### 功能清单

每日成长记录包含：

- 日期。
- 成长维度。
- 领域 / 科目 / 活动类型。
- 内容。
- 用时。
- 数量或强度，可选。
- 完成任务数。
- 未完成任务数。
- 专注度：很好、一般、容易分心。
- 难度感受：简单、适中、困难。
- 身体状态：精力充沛、正常、疲劳、不适。
- 情绪状态：开心、平静、抗拒、焦虑。
- 家长备注。
- 孩子自评。

#### 家庭版特色

家庭版应重视“成长状态”而不仅是“学习结果”。建议保留孩子自评、情绪、专注状态、身体状态和家长备注，避免产品变成单纯的监督工具。

### 5.5 领域、科目与能力点跟踪

#### 目标

帮助家长从“今天做了什么”进一步看到“哪些成长维度、科目、活动类型和能力点需要持续关注”。

#### 功能清单

- 支持维护成长维度：德育、智育、体育、美育、劳育。
- 支持维护领域 / 科目 / 活动类型：语文、数学、英语、科学、阅读、跑步、跳绳、游泳、绘画、音乐、家务、习惯等。
- 智育支持维护知识点。
- 体育、美育、劳育和德育支持维护能力点或习惯点。
- 支持记录掌握或养成程度：未开始、进行中、基本掌握、熟练、需要巩固。
- 支持关联任务。
- 支持智育关联错题。
- 支持查看最近练习次数、最近错误次数和最后复习时间。

#### MVP 范围

第一阶段只做家长手动维护领域、知识点、能力点和简单统计；不做完整 K12 知识图谱、专业体测系统、艺术等级评价、自动诊断、自动推荐和多教材全量覆盖。

### 5.6 错题 / 问题记录

#### 目标

建立智育维度下的高价值功能：错题记录、错因分析和复习提醒。错题不是整个系统的唯一核心，而是德智体美劳全面记录中的智育专项能力。

#### 功能清单

每条错题记录包含：

- 科目。
- 知识点。
- 题目照片。
- 孩子答案照片，可选。
- 正确答案，可选。
- 错因：概念不懂、粗心、审题错误、计算错误、记忆不牢、方法不会、时间不够。
- 是否已订正。
- 是否已复习。
- 复习提醒日期。
- 家长备注。
- 孩子自我解释。

#### 建议能力

- 支持 3 天、7 天、14 天复习提醒。
- 支持标记“已掌握”和“仍需复习”。
- 支持按科目、知识点、错因筛选。

#### 暂不做

- OCR 自动识别题目。
- 自动识别知识点。
- 自动批改。
- AI 解题。
- 题库匹配。

### 5.7 成长进度和周报

#### 目标

通过固定模板的周报，让家长低成本了解孩子本周在德智体美劳各维度的成长情况。

#### 周报内容

- 总学习天数。
- 总成长记录天数。
- 总投入时长。
- 任务完成率。
- 德智体美劳任务分布。
- 德智体美劳投入时长分布。
- 各科目学习时长。
- 各科目任务完成情况。
- 错题数量。
- 高频错因。
- 需要复习的知识点。
- 体育锻炼完成情况。
- 阅读和艺术练习情况。
- 劳动实践完成情况。
- 品德和习惯目标完成情况。
- 家长备注。
- 孩子自评。
- 下周建议。

#### MVP 范围

第一版使用固定模板即可，不需要 AI 生成和复杂报表。例如：

```text
本周学习总结：
- 共学习 5 天
- 完成任务 18/22
- 数学错题较多，主要集中在分数计算
- 英语单词复习完成较好
- 周三、周五专注度偏低
- 下周建议减少周五任务量，增加数学错题复习
```

调整后的家庭成长周报示例：

```text
本周成长总结：
- 共记录 6 天
- 完成任务 24/30
- 智育任务完成较好，数学错题集中在分数计算
- 体育锻炼完成 4 次，周末户外活动不足
- 美育练琴完成 3 次，主动性较好
- 劳动任务完成 2 次，整理房间仍需提醒
- 德育目标“按时睡觉”完成 4/7 天
- 下周建议保持数学错题复习，增加 2 次户外运动，固定周三和周六做劳动任务
```

### 5.8 家长反馈与孩子自评

#### 目标

将原学校场景中的“家校互动”收敛为家庭场景中的“亲子反馈”。

#### 功能清单

家长可以：

- 对任务写反馈。
- 对每日记录写反馈。
- 对每周报告写反馈。
- 给孩子发送鼓励语。
- 设置小目标。

孩子可以：

- 写一句自评。
- 选择今天心情。
- 标记“我需要帮助”。
- 给任务打难度标签。

#### MVP 范围

只做每日备注、每周反馈、状态选择和鼓励卡片；不做复杂聊天、群聊、会议或公告系统。

### 5.9 激励系统

#### 目标

避免产品变成单向监督工具，让孩子获得正向反馈。

#### 功能清单

- 完成任务获得星星。
- 连续打卡（第二阶段能力）。
- 每周小目标（第二阶段能力）。
- 徽章（第二阶段能力）。
- 奖励兑换。
- 家长手动确认奖励。

#### 设计原则

激励系统的目标是正反馈，不是游戏化成瘾。奖励应由家长可控，且与真实家庭约定结合。

第一阶段只实现“家长确认任务后获得星星、查看星星流水和余额、家长确认奖励兑换”。星星发放和兑换必须幂等并保留流水；连续打卡、每周目标和徽章进入第二阶段。

## 6. 第一阶段可以暂时不做的功能

### 6.1 学校 / 班级管理

暂不做：

- 学校组织。
- 班级。
- 教师加入班级。
- 学生批量导入。
- 年级和班级权限。
- 学校管理员。
- 班级排名。
- 班级统计。

原因：家庭版不需要组织结构，过早引入会显著增加权限和数据模型复杂度。

### 6.2 教师端完整功能

暂不做：

- 教师账号。
- 教师发布作业。
- 教师批改。
- 教师班级管理。
- 教师查看全班数据。
- 教师与家长沟通。
- 教师会议安排。

后续可扩展为家教或老师轻协作，但不进入第一阶段。

### 6.3 复杂家校互动 / 会议 / 视频会议

暂不做：

- 班级公告。
- 家长会安排。
- 视频会议。
- 师生群聊。
- 消息已读回执。
- 多人聊天。
- 复杂通知中心。

第一阶段只保留家长反馈、孩子自评、请求帮助和简单提醒。

### 6.4 复杂数据分析和个性化推荐

暂不做：

- AI 分析。
- 预测成绩。
- 大模型学习建议。
- 复杂雷达图。
- 学习路径自动规划。
- 推荐资源算法。
- 班级对比。
- 同龄人对比。

第一阶段只做简单统计和固定模板周报。

### 6.5 资源平台 / 内容库

暂不做：

- 公开资源市场。
- 资源推荐。
- 资源审核。
- 资源评分社区。
- 视频播放平台。
- 题库系统。
- 内容版权管理。

第一阶段只做资料上传、任务附件、错题图片和链接收藏等轻量能力。

### 6.6 微服务过度拆分

家庭版第一阶段不继续强化微服务拆分。建议优先保证：

- 能跑。
- 能测。
- 能部署。
- 能联调。

可以逻辑上保留模块边界，但实现和部署应尽量收敛，避免继续扩大服务数量和运维复杂度。

### 6.7 高级权限系统

暂不做：

- 学校管理员。
- 年级管理员。
- 班主任。
- 任课教师。
- 助教。
- 审核员。
- 多租户复杂权限。
- RBAC 后台配置。

第一阶段只需要家长和孩子两个核心权限边界。

## 7. 现有模块的家庭版映射

| 原模块 / 能力 | 家庭版重新定义 | 第一阶段处理建议 |
| --- | --- | --- |
| user-service | 家庭账号、家长、孩子档案 | 保留并收敛角色模型 |

## 11. 第一阶段执行边界

第一阶段只实现家长 Web 端和孩子简化入口。移动端仅做兼容性评估，不作为首轮验收入口。

第一阶段继续复用现有服务目录，但接口语义按家庭成长版重命名和收敛。学校、班级、教师、管理员、会议、公告、复杂资源库、AI 分析和复杂推荐均不进入首轮开发。

第一阶段的核心抽象是成长任务和成长记录，而不是学校作业。所有任务、记录、周报和提醒都必须支持成长维度：德育、智育、体育、美育、劳育。智育保留科目、知识点和错题专项能力，体育、美育、劳育和德育先使用活动类型、过程记录、家长反馈和孩子自评的通用结构。

## 13. 开发落地建议

### 13.1 先砍范围

第一版只围绕家庭版核心闭环实现，不继续扩大学校级功能范围。

必做：

- 家长。
- 孩子。
- 任务。
- 成长记录。
- 错题。
- 体育锻炼。
- 阅读、艺术、劳动和品德习惯。
- 周报。
- 激励。

暂缓：

- 学校。
- 老师。
- 班级。
- 家校会议。
- 复杂资源库。
- 复杂分析。
- 多租户后台。

### 13.2 先修工程基础

当前项目已有较多模块和测试，但仍需优先解决依赖缺失、路径引用错误、测试 mock 与真实实现不一致、接口契约不一致等问题。家庭版核心模块应优先形成稳定可运行的最小后端和前端闭环。

### 13.3 先做家长 Web 端

建议第一阶段优先实现家长 Web 端，然后再做孩子简化端和移动端优化。家长端管理功能最多，Web 端开发和调试效率更高。

### 13.4 先打通一个完整 Demo

第一个可演示闭环建议为：

1. 家长登录。
2. 添加孩子“小明”。
3. 给小明添加 5 个今日成长任务：数学练习、英语阅读、跳绳、练琴、整理房间。
4. 小明完成其中 4 个，并记录跳绳数量和练琴时长。
5. 家长确认完成。
6. 家长拍照记录 1 道数学错题。
7. 系统生成今日成长记录，覆盖智育、体育、美育和劳育。
8. 周末生成包含德智体美劳分布的成长周报。

该闭环顺滑后，再扩展模板、提醒、统计和移动端体验。
